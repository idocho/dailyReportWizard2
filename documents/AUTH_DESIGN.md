# 강사 인증·권한 설계 (#1 — 2차 Security, Firebase Auth + 화이트리스트)

**작성**: 2026-06-21 · **상태**: 설계(미구현) · **선행**: `SECURITY_RULES_PLAN.md`(1차 임시봉인)

> 1차(deny-by-default + 공유 DB Secret)는 **임시벽**. 이 문서의 2차가 완성되면
> 사용자 단말의 만능키(시크릿)를 회수하고 개인 로그인으로 교체한다.

---

## 1. 목표 (왜 하나)
- **퇴사자 차단**: 떠난 강사/관리자의 접근을 즉시 끊는다(계정 비활성·명단 삭제).
- **캠퍼스 격리**: A캠퍼스 강사가 B·C 데이터에 접근 못 하게 **룰로 강제**.
- **만능키 회수**: 사용자 단말에서 공유 시크릿 제거 → 단말 분실/유출=전체노출 위험 해소.
- **사용성**: 일반 강사는 가능한 무마찰(개인 기기 1회 인증 후 자동 유지).

## 2. 핵심 모델 — 개인 신원 + DB 화이트리스트(ACL)
- 모든 사용자는 **개인 신원으로 로그인**(Firebase Authentication).
- 권한은 DB의 **화이트리스트 노드(`acl/`)** 가 결정 — "이 신원 = 어느 캠퍼스·어느 역할".
- 룰은 `auth`(로그인 신원) + `acl/`(명단)만 보고 허용/차단. **앱 클라이언트 체크는 보조일 뿐, 차단은 항상 룰(서버)에서.**
- 커스텀 클레임(Admin SDK 서버 필요) 대신 **DB ACL을 룰이 직접 읽는** 방식 → 별도 백엔드 불필요, 관리자가 ACL만 관리.

```
acl/
  {uid}: { campus: "gangnam", role: "instructor"|"admin"|"super", email: "...", active: true }
    · 키 = Firebase Auth uid (불변·고유). 이메일 키는 회피(이메일 변경·'.'치환 취약).
      관리자 도구가 계정 생성 시 uid를 즉시 받으므로 uid 매핑이 자연스러움.
    · active:false 또는 노드 삭제 = 즉시 차단(퇴사). 토큰이 아직 유효해도 룰이
      매 요청 acl.active 를 확인하므로 즉시 효력(개인 계정의 강점).
    · email 은 표시·관리용 라벨일 뿐, 권한 판단엔 안 씀.
    · role: instructor / admin(캠퍼스) / super(전 캠퍼스 — §10 결정 시)
```

### 2.1 개인 DB 설정 제거 (로그인의 부수 이득)
로그인이 들어오면 사용자가 직접 넣던 DB 설정 3종이 모두 불필요해진다:
- **DB URL**: 앱 빌드에 **내장**(고정). Firebase DB URL은 비밀이 아니다 — 보안은 룰+auth가 담당(웹 Firebase 설정은 원래 공개되는 값).
- **경로/캠퍼스**: 로그인 후 `acl/{me}/campus` 를 읽어 **자동 판별** — 사용자가 고르지 않음.
- **DB Secret**: **제거** — 로그인 토큰이 대체. 단말에서 만능키가 사라져 유출면 동시 해소.

→ 강사 온보딩: "앱 실행 → 로그인"이 전부. 설정(`drw_db_url/path/secret`) 입력란은 **관리자 전용으로 축소 또는 제거**.
→ **단일 Firebase 프로젝트 확정**: 캠퍼스별로 프로젝트(창고)를 쪼개면 "어느 DB에 붙어 acl을 읽나"가 닭-달걀이 됨. 단일 프로젝트 + 캠퍼스는 acl/경로로 구분(§9 #3 해소).

## 3. 인증 방식 — **이메일+비밀번호 확정**
가장 보편·이해 쉽고, 개인기기(세션 유지)·공용 태블릿(앉을 때 입력) 양쪽 다 동작하므로
**이메일+비밀번호**로 확정. (매직링크는 공용기기에 매번 링크 클릭 부담, 익명+기기승인은 공용
태블릿에서 사람 구분 불가 → 모두 탈락.)

### 3.1 계정·비밀번호 관리 (관리자 생성형)
- **비밀번호는 Firebase Auth가 해시(scrypt)로 보관 — 관리자·앱 누구도 실제 비번을 보지 못한다.**
- **발급**: 관리자가 ClassManager(또는 웹 관리자)에서 계정 생성 + **임시 비밀번호** 지정 →
  강사에게 아이디+임시비번 **개별 전달**(비공개 채널).
- **첫 로그인 시 비번 변경 강제** → 이후 본인만 앎.
- **분실 처리(택1, §9에서 환경 따라 확정)**:
  - (a) **아이디=실제 이메일** → 강사 "비밀번호 찾기" 셀프 재설정(메일). 관리자 개입 0. **권장**.
  - (b) **이메일 미사용(내부 아이디 `id@캠퍼스.drw.local`)** → 관리자가 도구에서 비번 리셋 후 재전달.
- **퇴사**: `acl/{me}.active=false`(즉시 차단) + 필요 시 Auth 계정 비활성. 비번 회전 불필요(개인 계정).
- 계정 생성/삭제 UI = 관리자 도구. **개별 강사 비번은 관리자가 보관·관리하지 않는다**(해시+셀프리셋 구조).

## 4. 권한 룰 스케치 (경로별·역할별)
**캠퍼스 분리 = 경로 분리 확정**(`drw2_{campus}`). 단일+필드보다 룰이 훨씬 단순 —
서브트리마다 campus가 상수라 레코드별 검사·교차쓰기 방지가 필요 없다. `acl/`는
**루트(캠퍼스 경로 밖)** 에 둔다(로그인 시 campus 모른 채 먼저 읽어야 하므로).

```
// 헬퍼: 현재 사용자의 acl 엔트리 (uid 키)
me()        = root.child('acl').child(auth.uid)
active      = auth != null && me().child('active').val() == true
inCampus(C) = active && (me().child('campus').val()==C || me().child('role').val()=='super')
role        = me().child('role').val()

acl/
  ".read":  "role=='admin' || role=='super'"          // 명단 조회는 관리자/본사만
  acl/$uid ".read": "auth.uid==$uid"                   // 본인 엔트리는 본인도 읽기(campus 파악용)
  ".write": "role=='super' || (role=='admin' && newData.child('campus').val()==me().child('campus').val())"
            // 관리자는 자기 캠퍼스 강사만 추가/삭제, super는 전체

drw2_{campus}/                                          // 예: drw2_gangnam
  ".read":  "inCampus('gangnam')"
  students,classes 쓰기:        "inCampus('gangnam') && (role=='admin'||role=='super')"
  input,obs,scores,session 쓰기: "inCampus('gangnam')"   // 강사·관리자 공통(수업 운영)
  history 쓰기:                  "inCampus('gangnam')"   // PC 전송 경로
```
- 캠퍼스 격리 = `acl.campus == 경로 campus` 대조로 룰 강제(super는 전 캠퍼스 통과).
- 강사는 명단(students/classes) **읽기만**, 추가·삭제는 관리자(기존 v8.13 권한분리를 룰화).
- 강사 acl.active=false → **토큰 유효해도 즉시 차단**(룰이 매 요청 확인).
- (선택·후속) 캠퍼스 내 "담임 본인 수업만 쓰기"까지 룰로 좁히려면 배정맵 필요 —
  v1은 캠퍼스를 신뢰경계로 두고 캠퍼스 단위까지만 강제, 그 안은 클라 UI 가드 유지.

## 5. 관리자 계정 — 캠퍼스당 공유 1개 (비번 회전)
- 관리자 1~3명/캠퍼스 → **캠퍼스별 공유 관리계정 1개**(role=admin). 퇴사 시 **비밀번호만 변경**(단말 재설정 없음, 남은 관리자 다음 로그인 때 새 비번).
- 트레이드 ①: "어느 관리자가 했는지" 기록 안 남음 — 소수 신뢰 관리자라 수용.
- 트레이드 ②(솔직히): 공유 계정은 `acl.active=false`로 못 끊음(남은 관리자도 같이 막힘).
  퇴사 시 **비번 변경**에 의존 → 떠난 관리자의 기존 세션이 **최대 ~1시간(idToken 만료까지)**
  잔존(이후 refresh 차단). **즉시 차단이 필요하면 관리자도 개인 계정**으로 가야 함(그땐 acl.active 즉효).
- 이득: 개인별 관리계정 발급·관리 UI 불필요. (강사만 개인 계정 관리)
- ⚠️ 강사엔 공유 금지: 20명 공유 비번=사실상 유출 + 잦은 회전. 강사는 개인 계정 유지.

## 6. 클라이언트별 적용 (작업 규모순)
| 클라 | 사용자 | 적용 |
|------|--------|------|
| **웹앱(DRW)** | 강사 전원 + 관리자 | 🔴 본격 — 로그인 화면 + 세션 + 시크릿 제거. 핵심 |
| ClassManager | 관리자만 | 관리자 로그인 게이트 + acl 관리 UI(강사 계정 발급/삭제) |
| Analyzer | 관리자만 | 관리자 로그인 게이트(read-only) |
| PC앱(DRW) | 데스크/관리 직원(가정) | 관리자 로그인 또는 통제 PC 시크릿 유지. 강사 각자 운영이면 강사 로그인 추가 |

- **강사 계정 발급/삭제(=acl 관리)는 관리자 도구(CM 또는 웹 관리자)에서.** 이메일로 등록 → 강사 첫 로그인 시 매칭.

## 6.1 교차-클라 동일 계정 (웹 ↔ PC)
**같은 이메일+비번 계정 하나로 웹·PC 모두 로그인.** acl/룰은 클라 종류를 구분하지 않음 —
한 사람=한 신원. 시나리오는 동일하되 구현만 다르다:
- **웹앱**: Firebase JS SDK가 토큰 발급·1시간 갱신·세션 유지를 자동 처리.
- **PC앱(Python)**: REST로 직접 처리 — 공식 클라이언트 Auth SDK 없음. 필요 배관:
  1. 로그인 `POST identitytoolkit…/accounts:signInWithPassword?key={웹APIKey}` → `idToken`(1h)+`refreshToken`.
  2. RTDB REST 호출의 `?auth=` 를 **시크릿 → idToken** 으로 교체(`firebase.py _fb_url` 소폭 수정).
  3. idToken 만료 전 `POST securetoken…/token` (refresh_token)로 자동 재발급.
  4. **refreshToken 은 DPAPI 암호화 저장** — 기존 `secret_codec` 재활용(매번 재로그인 방지).
  - 필요한 키 = **Firebase 웹 API Key**(공개값, 비밀 아님).
- **퇴사 자동 반영**: 계정 비활성/비번 변경 → refreshToken 무효화 → 웹·PC 모두 다음 갱신 시 차단·재로그인 강제.

## 7. 만능키 → 로그인 무중단 전환 절차
1. Firebase Authentication 활성화(이메일/비번), 관리계정·강사계정 사전 발급, `acl/` 시드.
2. 룰을 **"시크릿 OR (auth+acl)"** 병행 허용으로 1차 배포 → 기존 시크릿 클라 안 끊김.
3. 클라이언트 순차 업데이트(로그인 탑재) + 사용자 로그인 전환 확인.
4. 전원 전환 확인 후 룰에서 **시크릿 우회 제거**(auth+acl만 허용).
5. Firebase 콘솔에서 **레거시 DB Secret 폐기**(회전) → 만능키 무효화. 백업 스크립트는 서비스계정/통제PC로 이관.

## 8. 단계별 작업 순서 (권장)
1. (선행) 1차 임시봉인 `deploy-rules.ps1` — 공개 차단하고 파일럿 출시.
2. 경로/캠퍼스 네이밍 확정(`drw2_{campus}`) + acl 스키마 확정.
3. Firebase Auth 활성화 + 룰 병행 버전 작성.
4. 웹앱 로그인 구현(최우선) → CM 관리자 게이트+acl관리 UI → Analyzer → PC.
5. 사용자 전환 → 시크릿 우회 제거 → 레거시 시크릿 폐기.

## 10. 설계 보강 — 엣지·보안 디테일 (검토 발굴)
- **G1 가입 선점 차단(중요)**: Firebase 이메일/비번은 클라가 임의 가입 가능 →
  공격자가 미래에 acl 등록될 이메일을 **미리 선점**하면 그 계정을 장악. **방어**:
  ① 관리자 도구가 **계정을 직접 생성**(uid 확보)한 뒤에만 acl 등록 — 아직 없는 이메일을
  acl에 넣지 않는다. ② 가능하면 셀프 가입 비활성. ③ acl이 최종 게이트라 acl 없는
  가입자는 어차피 접근 0(이메일 유니크 제약상 기존 계정 이메일 재가입 불가).
- **G3 acl 위치/읽기**: acl은 **루트**(캠퍼스 경로 밖). 본인 엔트리는 본인 읽기 허용
  (로그인 직후 campus 파악), 전체 명단 읽기는 관리자/super만(프라이버시).
- **G5 첫 관리자 부트스트랩**: acl 쓰려면 관리자, 관리자 되려면 acl — 닭/달걀.
  **전환 창에서 콘솔(또는 레거시 시크릿)로 최초 super/admin acl 1건 수동 시드.**
- **G6 브레이크글래스(잠금복구)**: 전환 중(시크릿 병행 단계)엔 레거시 시크릿이 비상문.
  시크릿 폐기 후엔 **Firebase 콘솔 룰 직접수정**이 유일 복구 → 프로젝트 owner 1인 상시 확보.
- **G7 오프라인 한계**: 로그인·토큰갱신은 네트워크 필요 → 무네트워크 시 접근 불가
  (acl 조회 불가). 동기화도 어차피 네트워크 필요하므로 수용. PC 진도 캐시는 읽기용으로만 유지.
- **G10 규모/비용**: Auth 사용자 ~70명(20×3+관리자) → 무료 한도(월 수만 MAU) 무관.
  룰의 acl 조회 비용 무시 가능. **비용·성능 우려 없음.**
- **G11 캠퍼스 내 범위**: v1은 캠퍼스가 신뢰경계 — 캠퍼스 안 강사 간 입력/관찰 상호접근은
  룰로 안 막고 클라 UI(담임 가드)만. 담임별 쓰기 제한이 필요해지면 후속(배정맵 룰).

## 11. 새 결정거리 (검토 중 발굴 — 확정 필요)
- **본사/슈퍼관리자(`super`)가 필요한가?** 3캠퍼스 사업이면 전 캠퍼스를 보는 오너/본사
  역할이 보통 필요(통합 분석·강사 이동·캠퍼스 신설). 필요하면 `role:'super'`로 룰에 이미
  반영해 둠(전 campus 통과). **불필요하면 super 제거하고 캠퍼스 admin만.** → 결정 요망.
- **Analyzer 범위**: 캠퍼스 admin = 자기 캠퍼스만 분석 / super = 전 캠퍼스 통합 분석.
  super 도입 여부와 직결.
- **관리자 즉시차단 필요성**: 공유 관리계정(~1h 잔존) vs 개인 관리계정(즉시) — §5 트레이드 택일.
