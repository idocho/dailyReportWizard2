# 플랫폼 재구성 — CampusManager 분리 + 전송 에이전트 (목표 아키텍처)

**작성**: 2026-06-21 · **상태**: 방향 결정(설계) · **연관**: `AUTH_DESIGN.md`(인증), `SECURITY_RULES_PLAN.md`

> 결정 배경: 강사·학생 계정/관리 기능을 관리자용 웹으로 모으고, 카톡 전송만 로컬로 남긴다.
> 인증(로그인)을 두 프로젝트가 공유하므로 계정 관리는 함께 간다.

---

## 1. 목표 컴포넌트 (역할 분리)

| 컴포넌트 | 형태 | 사용자 | 역할 |
|---|---|---|---|
| **DRW2 web** | 웹 | 강사 | 일일 리포트 입력·AI 생성·성적/진도 입력. **로그인만, 관리기능 제거** |
| **CampusManager** (신규) | 웹 | 관리자·super | **계정 관리**(강사 발급/비활성·역할·캠퍼스), 명단(students/classes) CRUD·CSV, 템플릿, 메시지 작성·수신자 선택·성적참조, **전송 작업 생성** |
| **전송 에이전트** (신규, CM에서 분리) | 로컬(윈도우) | 무인(캠퍼스 PC) — **관리자 도메인** | `sendJobs/` 큐 폴링 → KakaoTalk **일괄** 전송 → 상태 보고. **UI 없음**. 강사 일일 전송과 분리(§3) |
| **Analyzer** | 웹 | 관리자·super | 읽기 전용 분석(레이더·통계) |

- **CampusManager = DRW 웹의 관리자(adminOn) 기능 + ClassManager의 비전송 기능**을 흡수·격상.
- **전송 에이전트 = ClassManager의 카톡 전송 기능**만 떼어낸 얇은 로컬 실행기.
- 4개 모두 **동일 Firebase 프로젝트 + 동일 Auth/acl 공유**(AUTH_DESIGN).

## 3. 카톡 전송 — 도메인 분리 유지 (통합 안 함)
**운용 주체가 다르다.** 두 전송은 목적·주체·시나리오가 상이하므로 **합치지 않는다**:

| 전송 | 도메인 | 주체 | 목적 | 트리거 |
|---|---|---|---|---|
| **일일 리포트** | DRW(강사) | 강사 | 반별 학생 리포트 → 학부모 | 강사가 입력 후 발송 |
| **일괄 공지/성적** | 관리자 | 관리자 | 전체 공지·성적 일괄 | 관리자가 작성 후 발송 |

- **전송 에이전트(신규)는 관리자 도메인 전용** — CampusManager가 만든 일괄 작업만 실행.
- **DRW의 일일 리포트 전송은 강사 도메인에 그대로** — 기존 DRW 전송 경로(PC앱/강사측 전송)를
  유지. 관리자 에이전트로 끌어오지 않는다(주체·시나리오가 다름).
- 결과: 로컬 전송기가 **도메인별로 둘** 존재(의도된 분리). 같은 PC에서 동시에 카톡 창을
  다투지 않도록 운용만 분리(다른 시간/다른 PC).

```
강사 도메인:   DRW web(강사) ─ 작성 ─→ [DRW 전송(강사측 로컬)] ─→ KakaoTalk
관리자 도메인: CampusManager ─→ sendJobs/ ─→ [전송 에이전트(관리자)] ─→ KakaoTalk
```

> 미결: DRW 일일 전송의 로컬 형태(기존 PC앱 유지 vs 강사측 얇은 전송기로 정리)는
> DRW 슬림화(§5-3) 시 별도 판단. 단 **관리자 에이전트와는 분리** 원칙 고정.

## 4. 전송 에이전트 설계
- **신원**: 캠퍼스당 전용 **agent 계정**(acl `role:'agent'`, campus 한정). 자격은 로컬에
  **DPAPI 암호화 저장**(`secret_codec` 재활용). 헤드리스라 사람 로그인 부적합.
- **권한(룰)**: 자기 캠퍼스 `sendJobs/` 읽기 + 상태 쓰기, 방이름 위해 `students/` 읽기. 그 외 차단.
- **작업 흐름**: 큐에서 pending job pull → 카톡 전송(기존 `kakao_send.py` 로직 재사용) →
  각 건 성공/실패 상태 기록 → 웹이 진행 표시.
- **이미지**: 첨부 이미지 경로/데이터 전달 방식 결정(작은 이미지=base64 큐, 큰 건=별도 스토리지).

## 5. 마이그레이션 단계 (순서 — 변경폭 통제)
1. **인증(로그인) 완성** — 모든 앱의 토대(AUTH_DESIGN). CampusManager의 계정관리가 로그인 발급원.
2. **CampusManager 웹 — 계정관리부터** — 강사 계정 발급/비활성·역할·캠퍼스(로그인 전제). 이어서 명단·템플릿.
3. **DRW web 슬림화** — 관리자(adminOn) 기능을 CampusManager로 이관·제거. 강사 기능만 잔류.
4. **전송 에이전트 분리** — `sendJobs/` 큐 + 얇은 에이전트. **CM 일괄 전송만** 이관(관리자 도메인).
   DRW 일일 전송은 강사 도메인에 분리 유지(§3).
5. **구 ClassManager 은퇴** — 비전송 기능은 CampusManager로, 전송은 에이전트로 모두 이전 후.

## 6. 효용성
- 관리자 단일 웹 면(로그인 1회) — exe 배포·갱신 소멸.
- 강사 앱(DRW)은 관리코드 미적재 → **보안 경계 명확**(강사가 관리기능 코드 자체를 못 받음).
- 캠퍼스당 로컬 1개(전송 에이전트)로 축소 — 유지보수 격감.
- 계정·명단·템플릿 중앙화 + 룰 보호 + 백업.

## 7. 우려·판단
- **로컬 0 불가**: 카톡 자동화는 항상 로컬. 목표는 "최소 로컬화"(얇은 에이전트).
- **전송 PC 상시 가동**: 에이전트 돌고 PC 켜져 있어야 전송 발사. 새 실패모드(에이전트 다운=큐 적체) → 상태 모니터·알림 필요.
- **부품 수 증가**: 단일 CM → 웹+에이전트 2부품(+DRW). 디버깅·장애지점 증가.
- **범위 큼**: 인증 + 신규 웹앱 + 에이전트 분리 + DRW 슬림 = 플랫폼 재작성. **반드시 단계 분리.**
- **보안과 분리**: 이 재구성은 UX·유지보수 개선이지 보안 필수 아님(보안은 로그인이 해결). 우선순위는 로그인 다음.
